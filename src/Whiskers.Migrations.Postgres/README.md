# Whiskers.Migrations.Postgres

The **PostgreSQL** EF Core migrations for the app's two DbContexts (both in [`Whiskers.Data`](../Whiskers.Data/)) — a parallel assembly to [`Whiskers.Migrations.Sqlite`](../Whiskers.Migrations.Sqlite/), one model snapshot per context ([ADR-0004](../../docs/adr/0004-postgres-provider-support.md)).

- `Migrations/<timestamp>_InitialCreate` (+ `.Designer`) — `MetricsDbContext`, the provider-native PG schema: `bigint GENERATED BY DEFAULT AS IDENTITY` primary keys, `timestamp with time zone` for every `DateTime` (via the UTC converter), plus `text`, `boolean`, `double precision`. Independent of the SQLite baseline, so it carries its own timestamp id. `Migrations/MetricsDbContextModelSnapshot` is its snapshot.
- `Migrations/Identity/<timestamp>_InitialIdentity` — `WhiskersIdentityDbContext` (F1 local auth), the ASP.NET Identity user schema, applied at boot into its **own `__IdentityMigrationsHistory`** table. `Migrations/Identity/WhiskersIdentityDbContextModelSnapshot` is its snapshot.

The app pins this assembly with `MigrationsAssembly("Whiskers.Migrations.Postgres")` for both contexts.

Scaffold a new migration — the design-time factory branches on `WHISKERS_DB_PROVIDER`; `--context` is **mandatory** (two contexts share this assembly):
```
$env:WHISKERS_DB_PROVIDER="postgres"
# metrics
dotnet ef migrations add <Name> --context MetricsDbContext --project src/Whiskers.Migrations.Postgres --startup-project src/Whiskers
# identity (keep under Migrations/Identity)
dotnet ef migrations add <Name> --context WhiskersIdentityDbContext --project src/Whiskers.Migrations.Postgres --startup-project src/Whiskers --output-dir Migrations/Identity
```

To apply against a real PostgreSQL, the overlay [`deploy/docker-compose.postgres.yml`](../../deploy/docker-compose.postgres.yml) starts a server and points Whiskers at it. References `Whiskers.Data` only — never the app.
